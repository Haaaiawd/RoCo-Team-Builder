# ADR-003: 会话上下文管理策略（v2）

**状态**: Accepted
**日期**: 2026-04-07
**决策者**: 项目创始人

---

## 背景

`agent-backend-system` 需要在多轮对话中维护上下文。v2 延续“不持久化用户会话”的原则，但进一步明确：由于 `web-ui-system` 是基于 Open WebUI 的产品壳层，单用户可能同时打开多个聊天，因此仅用 `X-OpenWebUI-User-Id` 已不足以完全表达会话边界。

---

## 决策

**v2 使用内存 Session（非持久化），单进程部署（uvicorn `--workers 1`），会话 key 采用 `X-OpenWebUI-User-Id + X-OpenWebUI-Chat-Id` 组合键。**

- 主键规则: `session_key = "{user_id}:{chat_id}"`
- 缺失规则: 若 `user_id` 或 `chat_id` 任一缺失，**拒绝请求（400 Bad Request）**，不做回退。这消除了之前“回退到 user_id”导致的多聊天串上下文泄漏风险。
- 头部来源: `web-ui-system` 必须开启 `ENABLE_FORWARD_USER_INFO_HEADERS=true`，确保每个请求携带 `X-OpenWebUI-User-Id` 和 `X-OpenWebUI-Chat-Id`
- 生命周期: 随进程内存存在，进程重启或用户长时间无操作（30 分钟）后清除
- `ConfirmedOwnedSpiritList` 与历史消息一样，属于当前 `user_id:chat_id` 作用域下的短生命周期会话态；不持久化，不跨 chat 共享
- 上下文压缩: v2 继续不做自动压缩，超限时提示用户开启新对话

---

## 候选方案对比

| 方案 | 优势 | 劣势 | 结论 |
|------|------|------|------|
| **内存 Session + 组合键（严格模式）** | 满足不持久化要求，兼顾用户与聊天隔离，缺失时拒绝 | 不跨 worker，重启丢失 | ✅ 采用 |
| 仅 `user_id` | 实现最简单 | 同一用户多聊天可能串上下文 | 不采用 |
| `SQLiteSession` | 可恢复 | 违反“不持久化用户数据”原则 | 不采用 |
| Redis 自定义 Session | 支持横向扩展 | 当前阶段过度工程化 | 当前不采用 |

---

## 权衡点

### TP-1: 会话隔离粒度
- **风险**: 只按 user 维度隔离，会让同一用户不同聊天互相污染；若允许 chat_id 缺失回退，会在地址栏直接访问等场景下产生多聊天串上下文。
- **缓解**: 采用 `user_id + chat_id` 组合键，且**任一缺失时拒绝请求**，不做回退。

### TP-2: 单 worker 约束
- **风险**: 内存 Session 不跨进程共享。
- **缓解**: v2 明确保持单 worker 部署。

### TP-3: 长对话超限
- **风险**: 不使用 Responses Compaction，长对话可能超出模型窗口。
- **缓解**: 结果瘦身、回答简洁、超限时提示新开对话。

### TP-4: 为什么不持久化已确认拥有列表
- **风险**: 若把 `ConfirmedOwnedSpiritList` 升级为长期用户资产，会把一次截图识别确认误扩展成跨聊天、跨时段都生效的用户事实，增加数据最小化与状态过期风险。
- **缓解**: 继续把该列表定义为 chat 作用域的短生命周期会话态；重开聊天后需要重新确认。


## 后果

### 正面
- 满足 PRD 中“不持久化会话”的要求
- 同一用户多对话时隔离边界更精确
- 不引入额外基础设施依赖
- `ConfirmedOwnedSpiritList` 保持数据最小化，不会被误当成长期用户画像

### 负面
- 单 worker 约束继续存在
- 进程重启仍会丢失会话
- 用户重开聊天后需要重新确认截图识别得到的拥有列表

### 升级路径（v3+）
如需支持多 worker 或会话恢复，将 Session 后端替换为 Redis，自定义接口保持不变。
