# ADR-004: Web UI 裁剪与收敛策略

**状态**: Accepted
**日期**: 2026-04-07
**决策者**: 项目创始人

---

## 背景

本项目选择 Open WebUI 作为前端基座，原因是其具备成熟的对话 UI、BYOK、Rich UI 与部署能力。但 Open WebUI 本质上是通用 AI 平台，默认能力面大于本产品需求。如果不加控制，会出现以下问题：

1. 终端用户看到大量与配队产品无关的功能入口，产生认知噪音
2. 产品边界模糊，用户无法快速理解这是“洛克王国配队助手”而不是“通用 AI 门户”
3. 后续版本升级时，原生入口可能重新暴露，导致体验回退

---

## 决策

**将 `web-ui-system` 定义为“基于 Open WebUI 的受控产品壳层”，并采用“能力白名单”而非“默认全开”策略。**

同时明确三层资产关系，用于消除升级回流时的验收歧义：

1. **真理源**: `VisibleFeaturePolicy`
   - 定义终端用户可以看到什么、不能看到什么。
   - 是所有白名单判断、导航裁剪和发布验收的语义源头。
2. **唯一验证资产**: `VisibleFeaturePolicy` 导出的结构化快照/基线对象
   - 用于把白名单从“人工清单”提升为结构化验收输入。
   - 人工检查只能作为辅助手段，不能替代该资产。
3. **发布比较物**: 入口快照基线
   - 首页、侧边栏、顶部栏、设置面板、移动导航等 UI 入口必须与白名单导出快照对齐。
   - 若与基线不一致，则视为白名单回归未通过。

### 终端用户必须保留的能力
- 对话输入与消息展示
- 图片上传（用于精灵截图识别）
- BYOK 配置入口（面向愿意自带 Key 的用户）
- 内置模型轨道入口
- 工具调用折叠展示
- Rich UI 精灵卡片嵌入展示

### 终端用户必须隐藏、禁用或不可达的能力（类别级）
- 与本产品无关的通用知识库 / 工作区 / 文档管理入口
- 面向平台运营或开发者的构建、实验、调试、自动化入口
- 与配队问答无关的媒体生成或研究型入口
- 通用 AI 门户式的多功能导航结构
- 终端用户可见的后台管理与配置能力

### 管理员可保留但不得进入终端用户主路径的能力
- 必要的模型连接配置
- 部署与安全配置
- 上游基座维护入口

---

## 候选方案对比

| 方案 | 优势 | 劣势 | 结论 |
|------|------|------|------|
| **Open WebUI + 能力白名单裁剪** | 保留成熟底座能力，同时收敛产品边界 | 需要持续维护裁剪策略 | ✅ 采用 |
| Open WebUI 原样开放 | 成本最低 | 产品噪音大，用户心智混乱 | ❌ 不采用 |
| 完全自建前端 | 控制力最高 | 当前开发成本过高 | 当前不采用 |

---

## 权衡点

### TP-1: 上游复用 vs 产品独立性
- **风险**: 复用上游框架意味着需要接受一部分原生结构约束。
- **缓解**: 通过白名单策略与前端定制层，确保终端体验由产品目标主导。

### TP-2: 物理删除 vs 产品不可达
- **风险**: 某些上游能力可能难以彻底删除。
- **缓解**: 若无法物理删除，至少在终端用户路径上不可见、不可达、不可误触。

### TP-3: 升级回流风险
- **风险**: 上游版本升级可能重新暴露默认入口。
- **缓解**: 建立发布前 UI 回归检查，并要求其输入必须来自 `VisibleFeaturePolicy` 导出的结构化快照/基线；人工浏览清单只能作为补充，不能替代唯一验证资产。

---

## 后果

### 正面
- 产品边界更清晰，前端心智与垂类场景一致
- 减少用户认知负担，提高首次使用成功率
- `web-ui-system` 获得明确的设计与实现责任边界

### 负面
- 前端定制与回归测试工作量上升
- 升级 Open WebUI 时需要重新检查白名单约束

### 需要的后续行动
- [ ] 在 `web-ui-system` 设计文档中列出保留能力白名单与回归检查清单
- [ ] 将 `VisibleFeaturePolicy` 明确为白名单真理源，并定义其导出的结构化快照/基线为唯一验证资产
- [ ] 为终端用户路径建立首页、侧边栏、顶部栏、设置面板、移动导航等入口快照基线，并将其作为发布门槛
- [ ] 在实现阶段记录哪些能力通过配置关闭、哪些通过界面裁剪隐藏、哪些通过访问控制隔离
