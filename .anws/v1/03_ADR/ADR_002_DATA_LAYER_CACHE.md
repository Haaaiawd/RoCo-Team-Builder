# ADR-002: 数据层缓存策略

**状态**: Accepted
**日期**: 2026-04-07
**决策者**: 项目创始人

---

## 背景

`data-layer-system` 需要频繁访问 BWIKI MediaWiki API（每次工具调用均可能触发），存在以下风险：
1. BWIKI 无公开限流文档，过于频繁的请求可能被封禁或影响 BWIKI 服务
2. 相同精灵数据在同一对话内会被重复查询（如配队推荐过程中多次查同一精灵）
3. BWIKI 网络延迟不稳定，无缓存时每次工具调用都受网络影响

---

## 决策

**v1 使用 `cachetools` 内存 TTL Cache，不引入 Redis。**

缓存键：精灵名称（规范化后）+ 查询类型
缓存 TTL：精灵数据 **1 小时**，搜索结果 **10 分钟**
最大缓存条目：500 条（约覆盖游戏内全部精灵）

---

## 候选方案对比

| 方案 | 优势 | 劣势 | 适用场景 |
|------|------|------|---------|
| **`cachetools` 内存 TTL Cache** ✅ | 零额外依赖，零运维，代码内联配置 | 进程重启后缓存丢失，不跨进程共享 | v1 单进程部署 |
| Redis | 持久化，跨进程共享，TTL 精确 | 需要额外 Docker 容器，运维复杂度 ×2 | 多 worker / 多副本部署 |
| 无缓存 | 实现最简单 | 每次工具调用都打 BWIKI，延迟高、风险大 | ❌ 不可接受 |

---

## 权衡点

- **缓存击穿风险**：同一精灵被并发查询时可能同时穿透缓存打到 BWIKI。v1 单进程部署下并发量有限，用简单锁（`asyncio.Lock` per key）即可防护，不引入额外复杂度。
- **数据时效性**：BWIKI 精灵数据本身由社区维护，更新频率低（游戏版本更新时），1 小时 TTL 不会造成显著数据滞后。
- **内存上限**：500 条 × 每条约 2KB = ~1MB，对任何运行环境均无压力。

---

## 后果

### 正面
- Docker Compose 只需两个容器（Open WebUI + Agent 后端），部署极简
- `cachetools.TTLCache` 线程/协程安全，与 FastAPI asyncio 兼容
- 代码量极少，可测试性高（mock 缓存层即可）

### 负面
- 进程重启（如更新部署）后缓存清空，重启后首批请求延迟略高
- 单进程约束：不可横向扩展多 worker

### 升级路径（v2）
当需要多 worker 或持久化缓存时，将 `data-layer-system` 的缓存后端替换为 Redis，接口层不变，对 `agent-backend-system` 透明。Docker Compose 增加 `redis` 服务即可。
